  • Design Extensible Software Gang of Four

    I freaks out when coding becomes robotic-like chores. Here are two examples from both work and school. All of them have logical design and are used for many years. However, I do not like them because making changes involves tedious and repetitive work. The bad code smell here specifically is shotgun surgery.

    1. In order to add an additional attribute in an entity,
      1. Add it to the database table
      2. Add it to the Java entity class
      3. Add it to the Typescript entity class
      4. Add it to the Python entity class
      5. Add support in the manager class for this entity
      6. ......
      Why not create a library that generates cross-languages entities and serialization codes? So we only need to modify one file and the rest is auto-generated. In fact, thanks to Google, we have protobuf that does exactly this.

    2. In order to add support for new syntax in this interpreter,
      1. Add a new data type to represent this syntax
      2. Add an additional case in expression parsing phase
      3. Add an additional case in interpreting phase
      Why not create a interface called Syntax with two method parse() and interpret()? So we can group coupled procedures in the same place. I was facing similar issues in my Scheme Interpreter, and ended in another way.

    My point is that software design matters. A good design saves development time and headaches when requirement changes. I learned so much about design principals and patterns from Software design course of Dr. Hays. I want to work on extensible software that can grows, in particular libraries or frameworks used by a community. I want make impacts by applying design techniques to improve the productivity of the whole community.
